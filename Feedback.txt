수고 많으셨습니다. 전반적으로 무난하게 잘 진행 해주신 것 같습니다.
리드미를 잘 작성하시려고 노력하셨고, 커밋을 최대한 나눠서 잘 진행 하신 것 같습니다.
먼저 커밋은 지금처럼 잘 나눠서 하시면 좋은데, 메세지에는 다양한 컨벤션이 있습니다. 
이런 부분 찾아보시고 커밋 메세지도 조금 더 신경쓰시면 좋을 것 같습니다.

uri는 독립적으로, 역할에 따라 분리하려고 시도하신건 좋았습니다.
다만 계층적으로 불필요하게 "board"와 "feeds"가 중복되어있는 것 같습니다.
실제로 board 대신 feeds로 변경하고 나머지에서 feeds를 다 지우는게 더 깔끔하게 보이는 것 같습니다.

FeedResponseDto를 어떤 메서드에서는 사용하고, 어떤 메서드에서는 사용하지 않는건 
확실히 아쉬운 것 같습니다.

이 두가지만 수정하시면 요구사항적인 측면에서는 충분한 것 같습니다.
간단한 crud에 로직이 없는만큼 군더더기 없이 깔끔하게 작성해주신 것 같습니다.
이 아래에는 이후에 참고될 만 한 추가적인 리뷰 내용입니다.

주석은 꼭 명확한 이유를 가지고 작성하면 좋을 것 같습니다.
이상적으로는 변수명과 로직을 잘 분리해서 간단하게 만들면 필요 없기 때문에 달지 않는게 좋고
복잡한 로직이 꼭 필요하고 이해를 돕고 싶다면 메서드 앞에 차라리 더 자세하게 작성하는것이 좋은 것 같습니다.

에러 처리에 신경을 쓰시려고 했던 것 같아 보이는데, 
조금 더 잘 할 수 있을 것 같습니다.

어느 계층에서 에러를 처리할 것 인지, 
에러 이후의 조치(어떠한 정보를 ApiExceptionController로 흘려서 클라이언트에 보낼지, 어떠한 정보를 로그로 남길지)를 더 자세하게 구현 해 둘 수 있을 것 같습니다.


응답을 조금 더 일관적으로 만들 수 있을 것 같습니다.
많이들 사용하시는 ResponseEntity나 혹인 본인이 직접 모든 부문에서 사용하실 수 있는 Response전용 객체 (상태코드와 메세지를 기본적으로 담고있는)를 만들어보시는것도 방법입니다.
